## Morning (3-4h) — Phase F: SignalR Redis backplane

**Goal**: One-line code change enables multi-replica SignalR. Without backplane, a client connected to replica A misses events fired on replica B. With Redis backplane, all replicas share the SignalR event bus → any client receives all events regardless of which pod they're connected to.

Reference: [`scale-readiness-plan.md` Phase F (line ~517)](../scale-readiness-plan.md) + plan template at Phase F1.

### Context block — what's already in place

The Redis backplane has been **conditionally wired since P0** ([Program.cs:252-260](../../src/AMR.DeliveryPlanning.Api/Program.cs#L252-L260)) — gated by `SignalR:UseRedisBackplane=true`. Plus the package is already a project dependency. Today's job is **flipping the flag on + verifying multi-replica behaviour**, not writing new code.

```csharp
// Already in Program.cs — just needs SignalR__UseRedisBackplane=true env var
var useRedisBackplane = builder.Configuration.GetValue<bool>("SignalR:UseRedisBackplane");
if (useRedisBackplane)
{
    signalRBuilder.AddStackExchangeRedis(redisConn, options =>
    {
        options.Configuration.ChannelPrefix = StackExchange.Redis.RedisChannel.Literal("dtms:sr");
        options.Configuration.AbortOnConnectFail = false;
    });
}
```

### M3 — Multi-replica scale-up (~1h)

The compose file maps host port 5219 → container 8080 on a SINGLE service. Scaling api > 1 fails on port collision unless we either:
- remove the host port mapping (use only internal `api:8080`)
- use a load balancer (nginx / traefik / docker compose's built-in DNS round-robin)

**Cheapest path for dev**: scale via compose and use Docker's DNS round-robin from a sibling container (the headless test client) instead of the host port.

```powershell
# Scale api to 2
docker compose --env-file .env.test up -d --scale api=2 --no-recreate api
# (--no-recreate keeps existing dtms-api running, adds dtms-api-2)
# OR if compose complains about container_name: dtms-api conflict, remove container_name
# from the api service in docker-compose.yml (one-line edit) before scaling
```

If `container_name: dtms-api` blocks scaling — comment it out temporarily, or use the override pattern: keep `container_name` for normal flow, add a separate `api-scaled` service for the scale test only.

### M4 — Multi-replica broadcast test (~1.5h)

**The acceptance**: connect a hub client to replica A; trigger an event handled by replica B; verify the client on A receives the broadcast (proves Redis is fanning out).

Test script (extend `drain-handler-test.mjs` pattern):

```js
// scripts/chaos/backplane-test.mjs
// Two clients connected to potentially-different replicas via Docker DNS round-robin.
// Trigger a SignalR broadcast (e.g. via an admin endpoint that publishes to all clients).
// Assert: both clients receive the broadcast.
```

Concrete trigger: use an existing endpoint that produces a SignalR push. Options to consider:
- Submit an order via `/api/v1/delivery-orders/upstream` — eventually triggers OrderHub push (slow, requires consumer chain)
- Call `/admin/projection/republish/...` — direct projector replay (fast)
- Add a tiny test-only endpoint `POST /admin/signalr/broadcast` that emits to all hubs (anti-pattern but useful for the test; can ship gated to dev environment)

**Pragmatic call**: write the test against the existing `__drain` broadcast — already proven to fan out to all clients, no extra code needed.

```powershell

# Trigger drain on ONE replica only:
docker exec dtms-api curl -X POST -d '{"settleSeconds":3}' http://localhost:8080/api/v1/admin/drain-start

# Expected with backplane: BOTH clients receive __drain
# Expected WITHOUT backplane: only the client on the same replica gets it
```
