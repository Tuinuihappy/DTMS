### Outline

```
# K8s rollout runbook — DTMS API + outbox-worker

## Pre-rollout (one-time per cluster)
  - kubectl create namespace dtms
  - kubectl create secret generic dtms-secrets (DB password, JWT secret, RIOT key)
  - kubectl create configmap dtms-config (env vars from .env.test, NOT credentials)
  - Redis: shared cluster Redis OR per-namespace Redis deployment
  - RabbitMQ: shared cluster RMQ OR namespace-scoped
  - Postgres: existing managed instance OR per-namespace operator

## Initial rollout (cluster bootstrap)
  - kubectl apply -f deploy/k8s/dtms-config.yaml
  - kubectl apply -f deploy/k8s/dtms-secrets.yaml
  - kubectl apply -f deploy/k8s/postgres-migrator-job.yaml  (one-shot dotnet ef migrate)
    - wait for completion before api comes up
  - kubectl apply -f deploy/k8s/api-deployment.yaml   (from the .stub)
  - kubectl apply -f deploy/k8s/outbox-worker-deployment.yaml
  - kubectl apply -f deploy/k8s/service.yaml + ingress.yaml

## Verification (after initial rollout)
  - kubectl get pods -n dtms                        all Running + Ready
  - kubectl logs -n dtms deploy/dtms-api | grep "[Composition]"
    - confirm Vendor adapters chosen (NoOp vs Real)
  - kubectl exec -n dtms <pod> -- curl /health/ready  # 200
  - SignalR smoke: connect from a sidecar pod, verify __drain fan-out
    (use the drain-handler-test.mjs from scripts/chaos/)

## Rolling deploy (every subsequent version push)
  - docker push <registry>/dtms-api:<git-sha>
  - kubectl set image -n dtms deploy/dtms-api api=<registry>/dtms-api:<git-sha>
  - kubectl rollout status -n dtms deploy/dtms-api
    - K8s rolls one pod at a time:
      - new pod starts → readiness probe → join Service rotation
      - old pod preStop → drain-start → settle 25s → SIGTERM → exit
    - per-pod drain isolation (commit 5d4290d) means each step only
      affects clients on the draining pod, not the whole cluster

## Verify deploy success
  - kubectl rollout history -n dtms deploy/dtms-api
  - Check G2 shutdown logs from the old pods (kubectl logs -n dtms <old-pod>)
    - [Shutdown] total <Y.YY>s — should be < 60s per pod
  - Smoke: hit /api/v1/health/* from outside, run a quick chaos check

## Rollback (if rollout fails)
  - kubectl rollout undo -n dtms deploy/dtms-api
    - K8s does the same drain-aware rollback step by step

## Scaling decisions
  - api replicas: start 2; HPA on cpu > 70% target, max 5
  - outbox-worker replicas: start 1; scale up if outbox_pending > 5k for 5m
    - safe due to SKIP LOCKED (Phase A2)

## Disaster-recovery checks
  - drain endpoint loopback-only — confirm NetworkPolicy doesn't block
    intra-pod localhost calls
  - Redis pubsub TLS if cluster Redis requires it
  - terminationGracePeriodSeconds 90 covers the 25s settle + bus drain
    (currently MassTransitHostOptions.StopTimeout = 45s)
```
