## Morning (2-4h) — Step A5: Consumer parallelism

**Goal**: Tune MassTransit `ConcurrentMessageLimit` on the heavyweight consumers (Planning, Dispatch, projections) so a single consumer instance processes multiple messages in parallel. Expected drain rate uplift: 4-5×.

Reference: [`scale-readiness-plan.md` Step A5 (line ~190)](../scale-readiness-plan.md).

### Context block — topology decision (read first, decide before M2)

**Important**: Phase D split runs MassTransit consumers in BOTH `dtms-api` AND `dtms-outbox-worker` containers (worker = full app minus OutboxProcessor). RabbitMQ distributes via competing consumers → setting `ConcurrentMessageLimit=8` in code means **effectively 16 parallel handlers** across the system (8 per container).

Pick BEFORE tuning:
- **Option α** (recommended): set CMessageLimit per-consumer based on *per-container* desired concurrency. With 2 containers × CMessageLimit=4 = ~8 effective parallel handlers per heavy consumer.
- **Option β**: only api runs consumers (worker = OutboxProcessor only). Requires Track C (singleton service gating) first. Cleaner architecturally but bigger change.
- **Option γ**: blanket `cfg.ConcurrentMessageLimit = 8` global. Simplest but no per-consumer tuning.

This plan assumes **Option α** — tune per consumer, accept 2× effective concurrency.

### M2 — Apply tuning via ConsumerDefinition<T> classes (1h)

**Pattern**: codebase uses MassTransit auto-config (`cfg.ConfigureEndpoints(context)`). The right hook for per-consumer endpoint customization is a `ConsumerDefinition<T>` class — MassTransit auto-picks these up alongside the consumer.

```csharp
// Place next to the consumer it tunes (same namespace/folder)
public sealed class DeliveryOrderValidatedConsumerDefinition
    : ConsumerDefinition<DeliveryOrderValidatedConsumer>
{
    public DeliveryOrderValidatedConsumerDefinition()
    {
        // Per-container: 4. With Phase D 2-container split, effective = 8.
        ConcurrentMessageLimit = 4;
        // Raise prefetch to keep the parallel handlers fed.
        EndpointDefinition = new ConsumerEndpointDefinition<DeliveryOrderValidatedConsumer>(
            new EndpointSettings<IEndpointDefinition<DeliveryOrderValidatedConsumer>>
            {
                PrefetchCount = 32  // 4 × 8 buffer headroom
            });
    }
}
```

Apply to:
- `DeliveryOrderValidatedConsumer`         CMessageLimit=4 (heavy DB work, primary drain bottleneck)
- `TripStartedConsumer` + `TripCompletedConsumer` + `TripFailedConsumer` + `TripCancelledConsumer` + `TripPickupCompletedConsumer` + `TripDropCompletedConsumer`   CMessageLimit=4 each
- `TripStartedJobConsumer` + `TripCompletedJobConsumer` + `TripFailedJobConsumer` + `TripCancelledJobConsumer` + `TripPausedJobConsumer` + `TripResumedJobConsumer`   CMessageLimit=4 each
- Projection consumers   CMessageLimit=2 (lighter DB work)

**Don't** add ConsumerDefinition for: `CaptureFinalSnapshotConsumer`, `ShelfReleaseConsumer` (keep at default 1)

Build clean check.
