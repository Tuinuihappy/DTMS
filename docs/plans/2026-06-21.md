## Morning (2-4h) — Step A5: Consumer parallelism

**Goal**: Tune MassTransit `ConcurrentMessageLimit` on the heavyweight consumers (Planning, Dispatch, projections) so a single consumer instance processes multiple messages in parallel. Expected drain rate uplift: 4-5×.

Reference: [`scale-readiness-plan.md` Step A5 (line ~190)](../scale-readiness-plan.md).

### M1 — Audit current MassTransit endpoint configs (30 min)

```powershell
# Find every ReceiveEndpoint + ConfigureConsumer call
grep -rn "ReceiveEndpoint\|ConfigureConsumer\|ConcurrentMessageLimit\|PrefetchCount" src/ --include="*.cs" | grep -v Test
```

Catalog:
- Where each consumer endpoint is configured
- Current `PrefetchCount` value (T1.1 set 16 globally — verify)
- Current `ConcurrentMessageLimit` value (default 1 unless overridden)
- Which consumers do heavy DB work vs lightweight projection updates

Likely candidates for tuning (heavy):
- `DeliveryOrderValidatedConsumer` (Planning — primary target, drives most drain throughput)
- `TripDispatched`, `TripStarted`, `TripCompleted`, `TripFailed` (Dispatch lifecycle)
- `Riot3MissionAccepted`, `Riot3TaskFinished` (vendor callbacks)

Lighter candidates (projection writes, low DB impact):
- `OrderListViewProjector`, `OrderStatusHistoryProjector`, etc.

### M2 — Apply tuning per endpoint (1h)

Pattern:
```csharp
cfg.ReceiveEndpoint("delivery-order-validated", ep =>
{
    ep.PrefetchCount = 64;            // raise from 16 for parallel buffer
    ep.ConcurrentMessageLimit = 8;    // ← KEY: 1 → 8 parallel handlers
    ep.ConfigureConsumer<DeliveryOrderValidatedConsumer>(context);
});
```

Decisions per-consumer:
- Heavy (CMessageLimit=8): DeliveryOrderValidated, Trip lifecycle
- Medium (CMessageLimit=4): vendor callbacks
- Light (CMessageLimit=2): projections (still benefit from some parallelism)
- Default (CMessageLimit=1): keep unchanged

Build clean check.
