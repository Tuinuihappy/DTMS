### M3 — Switch to NpgsqlDataSource singleton (1.5h)

Pattern (per [`scale-readiness-plan.md:170-184`](../scale-readiness-plan.md#L170)):

```csharp
// Program.cs — replace ad-hoc UseNpgsql(connStr) calls
var npgsqlDataSource = new NpgsqlDataSourceBuilder(connStr)
  .EnableDynamicJson()
  .Build();
builder.Services.AddSingleton(npgsqlDataSource);

// Each DbContext (existing ConfigureNpgsql is reused):
services.AddDbContextPool<DeliveryOrderDbContext>((sp, o) =>
  o.UseNpgsql(sp.GetRequiredService<NpgsqlDataSource>(), ConfigureNpgsql));
```

8 DbContexts to migrate:
- `FacilityDbContext`
- `FleetDbContext`
- `DeliveryOrderDbContext`
- `PlanningDbContext`
- `DispatchDbContext`
- `AuthDbContext`
- `OutboxDbContext`
- `VendorAdapterDbContext`

**Caveat for `AddDbContextPool`** (must check each context):
- Pooled DbContexts MUST NOT take scoped services in the constructor (because pool reuses across requests).
- If a context takes scoped services → use lazy resolution via `IServiceProvider.GetRequiredService<T>()` in `OnModelCreating` or use a `IServiceProvider` injection pattern.
- If a context is genuinely stateful or constructor-heavy → keep on `AddDbContext` (non-pooled) and document why.

Build after each module switch to catch breakages early.

- **If M1 catalogs ≥ 1 DbContext that injects scoped services in constructor** — pooled migration would corrupt request state. Decision: either refactor the offending context (could be 2-3h on its own), or skip pooled migration for that context (document why, keep `AddDbContext` for it). Don't fight it — ship the migration for the contexts that are clean.
- **If M3 build break on `AddDbContextPool`** — most common cause is `OnConfiguring` overrides. Check each DbContext class for that, prefer the registration-time configuration only.
