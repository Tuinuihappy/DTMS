## What this proves

The plan called for n=100 confidence that T1 holds under kill-mid-pipeline stress: "inject 100 orders, kill API container at random delay 1-8s during each consumer execution; expect 0 stuck orders after 5-min settle". Until this run, T1 had only n=2 evidence (the original OD-0374 / OD-0375 recovery). This script provides the statistical confidence the plan demands.

### What this run proves about T1

Before this run, T1 had `n=2` evidence (the original OD-0374 / OD-0375 recovery). After: `n=100` with `19` randomised kill points + 5min settle + 100% success.

Every T1 component held under stress:

- **T1.1** MassTransit retry -- messages redelivered after each kill
- **T1.2** try/catch in consumer -- no exception leak left an orphan
- **T1.3** graceful shutdown -- 90s grace window let in-flight consumers drain
- **T1.4** watchdog -- backstop ran at least one tick across the settle window
- **T1.5** idempotency -- re-runs from MassTransit redelivery produced no duplicates
- **T1.7** admin replan -- not needed (no manual interventions)
- **T1.8** vendor guard -- not triggered (no orders advanced past vendor accept while being killed -- happy path stayed clean)

## How to reproduce

```powershell
# Full run (~20 minutes)
powershell -File scripts/chaos/kill-mid-pipeline.ps1

# Smoke test (~2-3 minutes)
powershell -File scripts/chaos/kill-mid-pipeline.ps1 -OrderCount 5 -KillEvery 2 -SettleMinutes 1

# Baseline (no chaos) — useful when comparing failure rate against a healthy pipeline
powershell -File scripts/chaos/kill-mid-pipeline.ps1 -OrderCount 100 -SkipChaos -SettleMinutes 2
```

The script is idempotent: every order ref is `CHAOS-<uuid>-<n>` so a second run never conflicts with the first. Old chaos orders accumulate in DB; clean up via the transactional pattern below (delete children explicitly because the schema lacks ON DELETE CASCADE on every projection table; we used this pattern on 2026-06-19 to clear 110 orders / 550 rows in ~150ms):

```sql
BEGIN;
DELETE FROM deliveryorder."OrderAuditEvents"
    WHERE "DeliveryOrderId" IN (
        SELECT "Id" FROM deliveryorder."DeliveryOrders"
        WHERE "OrderRef" LIKE 'CHAOS-%' OR "OrderRef" LIKE 'SIGTERM-%');
DELETE FROM dispatch."Trips"
    WHERE "DeliveryOrderId" IN (
        SELECT "Id" FROM deliveryorder."DeliveryOrders"
        WHERE "OrderRef" LIKE 'CHAOS-%' OR "OrderRef" LIKE 'SIGTERM-%');
DELETE FROM planning."Jobs"
    WHERE "DeliveryOrderId" IN (
        SELECT "Id" FROM deliveryorder."DeliveryOrders"
        WHERE "OrderRef" LIKE 'CHAOS-%' OR "OrderRef" LIKE 'SIGTERM-%');
DELETE FROM deliveryorder."OrderListView"
    WHERE "OrderId" IN (
        SELECT "Id" FROM deliveryorder."DeliveryOrders"
        WHERE "OrderRef" LIKE 'CHAOS-%' OR "OrderRef" LIKE 'SIGTERM-%');
```

Verify with no orphans left behind:

```sql
SELECT 'orphan_jobs'   AS check, COUNT(*) FROM planning."Jobs" j   WHERE NOT EXISTS (SELECT 1 FROM deliveryorder."DeliveryOrders" o WHERE o."Id" = j."DeliveryOrderId")
UNION ALL
SELECT 'orphan_trips',           COUNT(*) FROM dispatch."Trips" t   WHERE NOT EXISTS (SELECT 1 FROM deliveryorder."DeliveryOrders" o WHERE o."Id" = t."DeliveryOrderId");
```

## What the verdict means

**PASS** -- every chaos order reached a terminal or in-progress status (`Dispatched`, `InProgress`, `Completed`, `PartiallyCompleted`, `Failed`, `Rejected`, `Cancelled`). The T1 stack (MassTransit retry + graceful shutdown + watchdog + idempotency + vendor-acceptance guard) survived the kill-mid-pipeline stress without losing or stranding any order.

**FAIL** -- one or more orders ended up at `Planned` with no Trip and no vendor-order-key. That is the exact incident shape (OD-0374 / OD-0375) T1 was built to prevent; a fail here means we have a new gap to investigate before deploying to prod. The verdict prints the stuck refs for direct triage.

## Out-of-scope failures (don't count against T1)

- **Submit failures** during the kill window: expected (the api is briefly down between kill and restart). Re-submission is a client concern, not a T1 concern.
- **RabbitMQ `x-delayed-message` plugin missing**: makes `/health/ready` return Degraded for the `TripCancelledOmsNotify` endpoint. Doesn't affect the consumer + watchdog path. Track separately as RabbitMQ infra debt.

## What this test does NOT cover

- Load (`k6` against the API itself) -- a separate concern, deferred per plan section 8 prioritisation.
- Vendor-side reconciliation -- defense layer 2 in the T1.8 plan; not implemented yet.
- T2 Saga semantics -- saga flag is off during this run.
- **End-to-end completion** -- this is Phase 5 below.

## Phase 5 -- end-to-end completion verification (planned, not yet implemented)

The current script ends after verifying every order reached `Dispatched`. That confirms the **T1 stack** (consumer + crash recovery + vendor accept) works. It does **not** confirm the **full lifecycle**: vendor robot operate -> webhooks -> Trip status moves through `InProgress` -> `Completed` -> order cascades to `Completed`.

This came up after the 2026-06-18 N=100 run: we noticed RIOT3 had accepted 100 trips but only operated 4 of them within the next hour and then cancelled the other 96 (vendor-side expiration). The T1 cascade handled the cancellations correctly (96 orders moved Dispatched -> Cancelled via Phase b11 cascade), so the verdict still stood, but the observation surfaced a real gap.

### What Phase 5 would do

```
Phase 1 Setup
Phase 2 Chaos          <-- current scope
Phase 3 Settle
Phase 4 Verify (reached Dispatched)
   |
   | ... wait for vendor robot operate + webhooks ...
   |
Phase 5 (NEW)
   - poll every N minutes for trip status changes
   - timeout after CompletionTimeoutHours (default 24h?)
   - VERIFY: every chaos order reaches Completed | PartiallyCompleted | Failed
   - HARD FAIL: any order stuck at Dispatched | InProgress past the deadline
```

### Why it's deferred

- **Vendor scope**: RIOT3 robot dispatch + webhook delivery are outside T1. Phase 5 measures the integration, not the crash-recovery contract.
- **Dev pollution**: pumping 100 orders through the real RIOT3 endpoint per run leaves a queue of stale orders for the vendor's ops team to clean up. Phase 5 needs either (a) a stub vendor that auto-completes trips for us, or (b) a RIOT3 sandbox environment.
- **Runtime**: ~hours per run vs ~15 minutes. Not viable as a pre-commit gate.

### When this becomes worth building

- After T2 Saga ships: Phase 5 becomes the canonical verification for Saga state transitions across the full lifecycle.
- When a vendor stub / sandbox is available: replaces the dev-pollution concern.
- For pre-release verification before customer rollouts.

### Script changes that would be needed

- Add `[switch]$WaitForCompletion` and `[int]$CompletionTimeoutHours = 24` parameters.
- Replace Phase 4's verify SQL with a query for the additional terminal statuses.
- Loop with polling interval (5 min?) until timeout or all orders terminal.
- Emit progress every poll so a developer running the script can see it's working over a long window.

The current `Phase 4` verdict is left as the authoritative T1 contract. Phase 5 would print a separate "end-to-end completion" verdict that does not gate the T1 PASS/FAIL.
